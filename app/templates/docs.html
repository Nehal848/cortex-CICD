Cortex CI/CD Documentation

Comprehensive guide to deploying, configuring, and operating the Cortex Self-Healing CI/CD platform. Connect repositories, tune AI prediction models, and automate incident response end-to-end.

Overview

Cortex CI/CD is an AI-augmented, self-healing continuous integration and delivery platform built for modern cloud-native teams. It combines a real-time pipeline orchestrator, an ML failure-prediction engine, and automated remediation — all surfaced through a single unified dashboard.

AI-Powered Diagnostics
Predicts build failures before they occur using historical pipeline telemetry and ensemble ML models.
Autonomous Self-Healing
Automatically retries, reroutes, or rolls back failed deployments based on configurable healing strategies.
Kubernetes-Native
Deep integration with Kubernetes workloads — monitor pods, nodes, and services directly from the dashboard.
Intelligent Alerting
Noise-reduced alert routing via configurable severity thresholds, channels, and auto-silence policies.

Quick Start

Get Cortex running locally in under five minutes.

1
Clone the repository

Check out the source from GitHub and navigate into the project directory.

2
Create a virtual environment

Isolate dependencies using Python's built-in venv module.

3
Install dependencies

Install all required packages from requirements.txt.

4
Launch the server

Run the Flask development server — the dashboard is available on port 5000.

# 1. Clone
git clone https://github.com/CortexCICD/Self-Healing-CI-CD.git
cd Self-Healing-CI-CD

# 2. Virtual environment
python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate

# 3. Install deps
pip install -r requirements.txt

# 4. Run
python app/app.py
Default dev credentials: admin / admin. Change these before any production deployment.

Architecture

Cortex follows a modular monolith design. The Flask web layer handles HTTP routing and template rendering. Business logic — pipeline orchestration, self-healing decisions, and AI inference — lives in the agents/ and ml/ packages. Kubernetes integration is managed through the official Python client library. 

Self-Healing-CI-CD/
├── app/              # Flask web application
│   ├── routes/       # Blueprint route handlers
│   ├── templates/    # Jinja2 HTML templates
│   └── static/       # CSS, JS, assets
├── agents/           # Self-healing orchestration agents
├── ml/               # ML model training & inference
├── monitoring/       # Prometheus / Grafana configs
├── kubernetes/       # Helm charts & manifests
├── jenkins/          # Jenkinsfile pipeline definitions
└── docker/           # Dockerfile & compose configs

CI/CD Pipeline

The pipeline view provides a real-time overview of all runs across your connected repositories. Each run progresses through discrete stages: Clone → Build → Test → Deploy → Verify. Cortex injects a pre-flight AI health check between Build and Test to flag high-risk changes before they reach production.

Runs marked Healed indicate the self-healing engine intervened and successfully remediated a failure without human action.

Trigger a run manually from CI/CD Pipeline → Trigger Run, or via the REST API.

Self-Healing Engine

The engine monitors all pipeline stages and Kubernetes workloads continuously. When an anomaly is detected — crashed pod, failing test suite, or timed-out deployment — it selects the optimal remediation strategy based on failure type and severity score.

🔄
Auto Retry
Re-runs the failed stage up to a configurable max with exponential backoff.
⏪
Rollback
Reverts the deployment to the last known healthy image tag automatically.
📦
Pod Restart
Terminates and reschedules unhealthy pods across available nodes.
⚖️
Scale Up
Temporarily increases replica count when high load is the root cause.
🔀
Traffic Reroute
Shifts ingress to a canary or stable service while the primary heals.
🔔
Escalate
When automated strategies fail, creates a P1 alert and notifies on-call engineers.
Rollback is destructive for stateful workloads. Ensure PersistentVolume backup policies are configured before enabling automatic rollbacks on database services.

AI Prediction Engine

Cortex uses a gradient-boosted ensemble trained on historical pipeline telemetry to predict build failures up to 3 stages ahead. Features include code-change velocity, test-flakiness ratio, dependency delta, and infrastructure resource trends.

Risk Score
Every run receives a 0–100 risk score. Runs above the threshold (default 70) trigger pre-emptive diagnostics.
Root Cause Analysis
SHAP explainability surfaces the top contributing factors to any predicted failure in human-readable form.

Kubernetes Integration

Cortex connects to your cluster via the standard KUBECONFIG environment variable or an in-cluster service account. The Kubernetes view shows live pod status, node resource utilisation, and deployment history. 

# Point Cortex at your cluster
export KUBECONFIG=~/.kube/config

# Verify connectivity
kubectl cluster-info
For in-cluster deployments the service account needs get, list, watch, and patch on pods, deployments, and services.

Monitoring & Observability

Cortex exposes a /metrics endpoint compatible with Prometheus scraping. Pre-built Grafana dashboards in monitoring/ visualise pipeline throughput, MTTR, heal rates, and Kubernetes resource saturation.

Prometheus Metrics
Counters and gauges for pipeline runs, failures, heal events, and latency histograms.
Grafana Dashboards
Import ready-made JSON dashboards from the monitoring/ directory into your Grafana instance.

API Reference

All endpoints are served under http://<host>:5000. Authentication uses session cookies established via POST /login.

MethodEndpointDescription
GET/dashboardMain dashboard — telemetry summary and recent runs.
POST/api/pipeline/runTrigger a new pipeline run. Returns run metadata including ID and status.
GET/pipelineAll pipeline runs, statuses, and stage breakdowns.
GET/monitoringReal-time infrastructure and service health metrics.
GET/kubernetesKubernetes cluster workload status and node utilisation.
GET/predictionAI prediction dashboard — risk scores and contributing factors.
GET/self-healingSelf-healing audit log — events, strategies applied, and outcomes.
GET/alertsActive and historical alerts with severity and routing metadata.
GET/docsThis documentation page.
POST/loginAuthenticate and establish a session. Body: username, password.
GET/logoutTerminate the current session and redirect to login.

Configuration

Core settings are configured via environment variables. A .env file at the project root is loaded automatically at startup. 

# .env — Application configuration

FLASK_SECRET_KEY=your-production-secret-key
FLASK_ENV=production

# Kubernetes
KUBECONFIG=/etc/cortex/kubeconfig

# ML model
ML_MODEL_PATH=ml/models/failure_predictor.pkl
ML_RISK_THRESHOLD=70

# Self-healing policy
HEALING_MAX_RETRIES=3
HEALING_BACKOFF_SECONDS=30
HEALING_ENABLE_ROLLBACK=true

# Alerting channels
ALERT_SLACK_WEBHOOK=https://hooks.slack.com/services/...
ALERT_EMAIL_SMTP=smtp.yourdomain.com
Never commit .env to version control. Add it to .gitignore and use your CI/CD secrets manager for production deployments.
